# Self-Aware Briefing — память + новости

Дата: 2026-06-14
Статус: дизайн утверждён, готов к плану реализации

## Цель

Дать торговому агенту (Claude) больше релевантного контекста при принятии
решений, чтобы решения были обоснованнее и приносили результат:

1. **Память** — агент видит свой собственный прошлый опыт: какие сделки открывал,
   с каким обоснованием и чем они кончились (реализованный/нереализованный P&L),
   плюс агрегаты (win-rate, доходность vs SPY). Обучение на своих ошибках.
2. **Новости** — свежие заголовки по торгуемым символам как внешний контекст.

Обе подсистемы независимы и подключаются в одну точку — `build_briefing`.
Гардрейлы, валидационная панель и исполнение **не меняются**.

## Принятые решения

- **Вариант A («самоосознающий брифинг»)**: обогащаем брифинг детерминированными
  данными, Claude сам делает выводы. Без второго вызова LLM (шаг рефлексии —
  отдельная будущая надстройка B). Без платных внешних API (вариант C отклонён).
- **Источник новостей**: `yfinance Ticker.news` — бесплатно, уже в зависимостях,
  никакой новой точки отказа на VPS.
- **Проверка эффекта**: прогон `simulate.py` (baseline vs с памятью) с метриками
  из `analysis/track_record.py`.

## Архитектура

```
fills + decisions ─► round_trips() ──► memory block ┐
positions + цены  ─► unrealized P&L ────────────────┤─► Briefing ─► prompt ─► Claude
news source       ─► заголовки по символам ─────────┘
                                                      ▼
                          (guardrails / panel / execution — без изменений)
```

`Briefing` остаётся единственным, что видит модель. Память и новости — два
отдельных, независимо тестируемых блока, собираемых внутри `build_briefing`.

## Компоненты

### 1. `analysis/round_trips.py` (новый, чистые функции)

FIFO-сопоставление открывающих и закрывающих сделок из таблицы `fills` для
одного агента. Возвращает список закрытых round-trip'ов:

- `symbol`, `quantity`
- `entry_date`, `entry_price`, `exit_date`, `exit_price`
- `realized_pnl` (абсолютный и %)
- `rationale` — исходное обоснование открывающей сделки (join по `decision_id`
  → `decisions.rationale`)

Поддержка long и short (знак qty). Частичные закрытия матчатся FIFO. Этого
расчёта в проекте сейчас нет — `reporting/queries.py` считает только
нереализованный P&L по открытым позициям и P&L по кривой equity.

### 2. Блок памяти в `Briefing`

Расширяем dataclass `Briefing` и `build_briefing` (прокидываем `journal`):

- `open_positions_memory`: по каждой держимой позиции — исходное обоснование
  (из самой ранней ещё открытой открывающей сделки по символу) + текущий
  нереализованный P&L %.
- `recent_closed`: последние ~10–15 round-trip'ов с реализованным P&L и
  обоснованием.
- `self_stats`: win-rate, средний выигрыш/проигрыш, суммарная доходность,
  сравнение с SPY за период трек-рекорда.

Холодный старт (нет истории) → блоки пустые → поведение идентично текущему.

### 3. `data/news.py` (новый)

- Протокол `NewsSource` с методом `headlines(symbol, as_of_date) -> list[Headline]`.
- `YFinanceNews` — реальная реализация через `yf.Ticker(symbol).news`.
- `FakeNews` — детерминированная реализация для тестов и офлайна.
- `Headline`: `title`, `publisher`, `published_date` (link опционально).
- Ограничение объёма: ~3–5 заголовков на символ, только held + universe.
- **Graceful degradation**: любая ошибка/таймаут → пустой список, цикл не падает
  (как resilience в боте/`httpx`).
- Переключатель `NEWS=yfinance|fake` в env, по образцу `BROKER`/`STRATEGY`.

### 4. Промпт (`agent/prompts.py`)

- `build_user_prompt`: рендерит блок памяти (открытые позиции с обоснованием+P&L,
  последние закрытые сделки, сводка) и блок новостей.
- `build_system_prompt`: добавляем инструкции — учиться на убыточных сделках;
  взвешивать новости, но **не выдумывать** заголовков, которых нет в списке, и не
  реагировать на устаревшие. (Инструкции в промпте — на английском, как код;
  поле rationale модель по-прежнему пишет по-русски.)

### 5. Проводка (`run.py`, `orchestrator/cycle.py`)

Прокидываем `journal` и `news_source` в `build_briefing`. Реальные/фейковые
реализации выбираются по env, как существующие `BROKER`/`STRATEGY`/`NOTIFIER`.

### 6. Замер (`orchestrator/simulate.py`)

Прогон на истории: baseline vs с-памятью. Сравнение P&L / Sharpe / max drawdown
через готовые функции `analysis/track_record.py`. Это доказательство, что
контекст реально двигает результат, а не вера.

## Честное ограничение

`yfinance .news` отдаёт **текущие** заголовки, без point-in-time доступа к
историческим новостям на произвольную дату. Следствие:

- **Память** проверяется на историческом бэктесте корректно (всё из журнала
  point-in-time).
- **Эффект новостей** на бэктесте измерить нельзя — это был бы look-ahead.
  Поэтому в бэктесте новости **отключаем**; их влияние оцениваем только вперёд,
  в paper-режиме.

## Обработка ошибок

- Новости: обёрнуты в таймаут + try/except → пустой список. Никогда не валят цикл.
- Память: при отсутствии истории блоки пустые (холодный старт = текущее поведение).
- Бэктест: новости принудительно отключены (нет look-ahead).

## Бюджет токенов

- Новости: ~3–5 заголовков на символ.
- Закрытые сделки: последние ~10–15.
- Держим в пределах `MAX_TOKENS` контекста агента.

## Тестирование

- Юнит-тесты FIFO-матчинга round-trips (long/short, частичные закрытия).
- `FakeNews` детерминированная.
- Снапшот-тесты рендеринга брифинга (память + новости).
- Тест холодного старта (нет истории → пустые блоки → поведение как сейчас).
- Тест graceful-degradation при падении источника новостей.

## Вне скоупа

- Шаг рефлексии (второй вызов LLM, вариант B) — будущая надстройка поверх A.
- Платные внешние данные / сентимент-скоринг (вариант C).
- Изменения гардрейлов, валидационной панели, исполнения, управления стопами.
- Историческая база новостей с point-in-time доступом.
