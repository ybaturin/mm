# LLM Edge Measurer (PEAD) — Design Spec

**Дата:** 2026-06-14
**Статус:** утверждён к реализации
**Автор:** Iurii Baturin

## 1. Цель

Измерить, есть ли у LLM **предсказательная сила** на earnings-событиях — *до* единого
реального доллара. Инструмент отвечает на один вопрос: «обыгрывает ли глубокое чтение
первоисточников рынок (SPY) после издержек, или это самообман?».

Это **не торговая система**. Это отдельный исследовательский модуль, который не трогает
торговый цикл. Его вывод — гейт: строить ли дальше торговлю на этом источнике альфы.

### Контекст решения

Базовый разбор проекта показал: риск-контур (guardrails, watchdog, validation panel)
сделан отлично, но **источник альфы — слабое звено**. Daily news-trading структурно
мёртв: к pre-market публичная новость уже в цене. Edge не появляется от «умной модели с
высоким эффортом» — рынок обыгрывают информацией/скоростью/масштабом/аномалией, а не
качеством рассуждения о публичных данных.

Единственный кандидат с **документированной** аномалией — **PEAD**
(post-earnings-announcement drift): рынок медленно отыгрывает качественные нюансы,
зарытые в длинных первоисточниках (тон Q&A на earnings call, формулировки гайденса).
Здесь узкое место — глубина и объём чтения, а его и чинит топ-модель с высоким эффортом.
Прежде чем перестраивать торговлю под это — измеряем, есть ли сигнал вообще.

## 2. Принципы

- **Измерить, потом торговать.** Ни доллара реальных денег, пока edge не доказан.
- **Честность важнее красивого результата.** Дизайн заточен на то, чтобы поймать
  собственный самообман (lookahead, бета, accuracy-обманка), а не на то, чтобы получить
  обнадёживающую цифру.
- **Бесплатно на старте.** Лин-пилот без платных подписок. Платный API — осознанная
  инвестиция только *после* положительного пилота.
- **Разовый батч, не вечный жор.** Измеритель — bounded-эксперимент (~5M токенов), не
  ежедневный поток. Это снимает вопрос «$/день» на этом этапе.
- **Переиспользуем существующее.** Point-in-time дисциплина, `MarketDataSource`,
  strict structured output, БД-журнал — паттерны уже есть в проекте.

## 3. Центральная проблема — lookahead bias и как мы его обходим

**Решения LLM нельзя бэктестить на истории** (исходный спек, §11): модель помнит исход
для любой даты до обучения, «предсказание» становится воспоминанием, бэктест даёт
фиктивный результат.

**Обход (подход A).** Окно между **knowledge cutoff** модели (~январь 2026) и **сегодня**
(июнь 2026) — это бесплатный, честно out-of-sample отрезок. Для earnings, случившихся
*после* cutoff, модель физически не знает исхода: на момент её обучения этого будущего не
существовало. Lookahead тут не работает.

- **Тестовая выборка:** только earnings-события с датой строго после knowledge cutoff
  модели (с запасом-буфером в несколько недель от cutoff, чтобы исключить пограничные
  утечки).
- **memory-probe (страховка):** перед запросом прогноза — отдельный дешёвый вызов:
  «знаешь ли ты фактический исход для этой акции после этого отчёта?». Если модель
  демонстрирует знание исхода — событие **исключается** из выборки.
- **Переход A→C:** то же хранилище и пайплайн естественно копят форвард-выборку дальше
  (новые события дописываются по мере выхода). A даёт немедленный предварительный ответ;
  C наращивает чистую выборку со временем. Это один инструмент, не два.

**Ограничение подхода A:** окно короткое (~4–5 мес), выборка ограничена. Честный исход
пилота может быть «недостаточно данных для вывода» — это валидный результат, не провал.

## 4. Что читается (вход модели)

Всё — **point-in-time**: только документы и цены с датой ≤ дня решения (день после
выхода отчёта).

| Документ | Что это | Источник | Роль |
|---|---|---|---|
| **Earnings call transcript** | стенограмма квартального звонка: prepared remarks + Q&A аналитиков | free-tier API / скрапинг | **главный сигнал** — тон, заминки, формулировки гайденса |
| **8-K (пресс-релиз)** | официальное объявление результатов (exhibit 99.1) | SEC EDGAR | твёрдые цифры |
| **10-Q (MD&A)** | «обсуждение менеджментом» + изменения риск-факторов | SEC EDGAR | детали, риски |
| **Цены до дня решения** | контекст + расчёт фактического форвард-дохода | yfinance | оценка |

Транскрипт — ядро тезиса: цифры рынок отыгрывает за секунды (они в 8-K, видны всем),
а качественный сигнал Q&A переваривает днями. Для лин-пилота допустимо начать **только с
транскрипта** (дешевле по данным); 8-K/10-Q добавить, если есть проблеск.

## 5. Прогноз (выход модели)

Strict structured output (как в Agent Core основной системы):

| Поле | Тип | Описание |
|---|---|---|
| `direction` | up / down / neutral | направление **market-adjusted** движения |
| `magnitude` | float (%) | ожидаемое движение сверх рынка |
| `confidence` | float 0–1 | уверенность модели |
| `horizon` | const = 5 торг. дней | окно дрейфа (см. §5.1) |
| `rationale` | str | обоснование (для разбора, **в метрику не идёт**) |

### 5.1 Почему горизонт = 5 торговых дней

- **1 день** — слишком близко к мгновенной реакции на цифры; ловит то, что уже в цене.
- **20 дней** — слишком далеко; за месяц сигнал тонет в новом потоке новостей и
  макро-шуме, причинность размывается.
- **5 дней** — окно, на котором классически наблюдается PEAD-дрейф: рынок «дочитывает»
  качественный нюанс. Это стартовое значение; измеритель считает метрики и для
  альтернативных горизонтов (1/5/20) как диагностику, но гейт — на 5.

## 6. Метрики edge

`accuracy` («угадал направление») **запрещена как основная метрика** — она врёт:
бета загрязняет (рост на бычьем рынке ≠ edge), игнорирует размер (9 угадал по +0.1%, 1
проспал −5% = «90%» и убыток), не учитывает уверенность.

Все доходности — **market-adjusted** (минус SPY за то же окно).

| Метрика | Что отвечает | Ориентир |
|---|---|---|
| **Information Coefficient (IC)** | ранговая корреляция прогноза с фактом | >0.05 устойчиво — уже хорошо |
| **Long-short спред** | сортировка по сигналу, «лонг топ / шорт дно», спред **после издержек** | это и есть «сколько бабла» |
| **Calibration по confidence** | растёт ли точность с ростом уверенности модели | монотонный рост = торгуемо |
| **Стат-значимость** | t-stat / p-value с поправкой на размер выборки | не шум ли вообще |

**Обязательные бенчмарки** (гейт честности):
1. **Монетка** (50/50 случайно) — нижняя граница.
2. **Тупой механический PEAD** — «купить всех, кто превысил консенсус по EPS» (earnings
   surprise по цифрам, без чтения). Если LLM не бьёт даже это — глубокое чтение не нужно,
   платить за транскрипты незачем.

Вывод измерителя по каждой метрике: значение, бенчмарки, значимость, размер выборки.

## 7. Архитектура

Отдельный модуль (напр. `src/trading/edge/` или `src/research/`), изолированный от
торгового цикла. Переиспользует инфраструктуру проекта:

- **Point-in-time дисциплина** — требование «отдавать данные как на тот день» (исходный
  спек §11). Тот же принцип на входе модели.
- **`MarketDataSource`** — реализация исторических цен (yfinance) для расчёта факта.
- **Strict structured output** — тот же подход валидации схемы, что в Agent Core.
- **БД-журнал** — таблица `edge_predictions` (событие, hash входа, прогноз, факт,
  метрики, флаг memory-probe). Это и есть будущий форвард-журнал (A→C бесплатно).

### Поток

```
1. Select events    — earnings после cutoff, из юниверса, с буфером
2. Collect docs     — транскрипт + 8-K + 10-Q + цены, всё point-in-time ≤ дня решения
3. Memory-probe     — отсеять события, где модель знает исход
4. Predict          — LLM, deep read, strict schema → прогноз
5. Realize          — фактический market-adjusted forward return (горизонт 5д)
6. Score + report   — IC, long-short, calibration, значимость vs бенчмарки
```

Шаги детерминированы и тестируемы по отдельности (как guardrails в основной системе).

## 8. Юниверс и выборка

- **Старт:** существующий ликвидный large-cap юниверс (`config/universe.toml`) —
  для них транскрипты доступнее всего.
- **Масштаб пилота:** ~100 событий (лин), ~5M токенов суммарно, разовый прогон.
- **Улучшение после пилота:** расширение в small/mid-cap, где PEAD **сильнее** (меньше
  аналитического покрытия → рынок медленнее переваривает), но данные сложнее.

## 9. Данные (бесплатно под пилот)

| Данные | Источник | Заметка |
|---|---|---|
| Транскрипты | free-tier API (FMP / API Ninjas) + точечный скрапинг Motley Fool для дыр | ~100 вызовов влезает в бесплатный лимит за день-два; для разового пулла фрагильность скрапинга приемлема |
| 8-K / 10-Q | SEC EDGAR | бесплатно навсегда |
| Цены, даты earnings | yfinance | уже в проекте |

**Платный API (FMP ~$20–50/мес) — улучшение после положительного пилота**, не ставка
вслепую. Если пилот покажет проблеск, чистые данные оправданы.

## 10. Бюджет токенов

| Статья | Токены/событие |
|---|---|
| Вход (транскрипт + куски филингов) | ~25–40k |
| Рассуждение (высокий эффорт / thinking) | ~5–15k |
| memory-probe | ~2k |
| **Итого** | **~40–55k** |

- **Лин-пилот:** ~100 событий ≈ **~5M токенов**, разово (гоняется за вечер).
- **Полный:** ~300 событий ≈ ~15M, если пилот обнадёжит.

Ручки управления объёмом: размер юниверса, глубина документов (только транскрипт vs
+ филинги), число проходов на событие.

## 11. Критерии успеха (что значит «измеритель работает»)

- Пайплайн прогоняет ~100 post-cutoff событий end-to-end без ручного вмешательства.
- Каждый шаг (выбор, сбор, probe, прогноз, оценка) покрыт тестами и детерминирован там,
  где это возможно.
- На выходе — отчёт с IC, long-short после издержек, калибровкой, значимостью и сверкой
  с монеткой и тупым PEAD, с явным размером выборки.
- Lookahead закрыт: только post-cutoff события + memory-probe фильтр, факт считаем мы.
- Результат интерпретируем как гейт: «edge есть / edge нет / данных мало».

## 12. Гейт-решение по итогам пилота

| Исход пилота | Решение |
|---|---|
| LLM бьёт тупой PEAD и значимо > монетки, long-short > 0 после издержек | строить торговый контур на этом источнике; оправдан платный API и расширение выборки |
| LLM ≈ тупой PEAD или ≈ монетка | глубокое чтение не даёт edge; не вкладываться, искать другой источник |
| Недостаточно данных (короткое окно) | переключиться на форвард-накопление (C), решение отложить |

## 13. Вынесено за рамки (YAGNI)

- Реальная торговля и любое исполнение — отдельный следующий этап, только при
  положительном гейте.
- Платные данные — после пилота.
- Small/mid-cap расширение — улучшение после пилота.
- Несколько моделей / ансамбль валидаторов — не для первого замера.
- Оптимизация горизонта/параметров под результат — запрещена на пилоте (это путь к
  переобучению на короткой выборке).
