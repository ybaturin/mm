# IBKR Trading Agents — Design Spec

**Дата:** 2026-06-13
**Статус:** утверждён к реализации
**Автор:** Iurii Baturin

## 1. Цель

Автономная система из трёх торговых агентов на счёте Interactive Brokers (IBKR),
которая ежедневно анализирует рынок, предлагает и исполняет сделки с целью извлечения
прибыли — под жёстким контролем рисков и с полной отчётностью владельцу.

Ключевой принцип проекта: **LLM считается ненадёжным компонентом.** Он только
предлагает сделки; исполняет их детерминированный код, который агенту не доверяет.

## 2. Принципы и ограничения

- **Сначала бумага.** Вся система строится и валидируется на paper-счёте IBKR.
  Реальные деньги подключаются последним шагом, сменой одного конфига.
- **Защита слоями.** Самые сильные ограничения вынесены за пределы нашего кода
  (на сторону брокера), потому что в коде может быть баг.
- **Безопасность важнее доходности.** Защитный пол (kill-switch) не отключается
  никогда, в том числе у агрессивного профиля.
- **Реалистичные ожидания.** Базовое ожидание доходности — «как рынок или хуже».
  Цель в 100%/год отвергнута как несовместимая с сохранностью капитала; вместо
  обещаний — измерение результата на симуляции.
- **YAGNI.** На старте без ML-модели и без второго LLM-валидатора.

## 3. Три агента

Все три работают раз в день перед открытием рынка (pre-market). Отличаются только
лимитами. Инструменты: ликвидные US-акции и ETF. Шорты — только агрессивному, и
только с обязательным стоп-лоссом.

| Профиль | Бюджет (paper) | Макс. на 1 позицию | Мин. позиций | Шорты | Стоп-лосс | Сделок/день |
|---|---|---|---|---|---|---|
| Консервативный | $5k | 15% | 8 | нет | −8% | 2 |
| Умеренный | $5k | 25% | 5 | нет | −10% | 4 |
| Агрессивный | $5k | 40% | 3 | да (со стопом) | −12% | 8 |

Все числа конфигурируемы.

### Виртуальные суб-счета

Три агента делят один физический счёт IBKR, но система ведёт для каждого
виртуальный суб-счёт: свой бюджет, свои позиции, свой P&L-журнал. IBKR видит один
счёт; внутри система знает, какая позиция какому агенту принадлежит. Так лимиты и
отчёты считаются раздельно.

## 4. Архитектура

```
VPS (Linux, 24/7)
  Scheduler (cron, pre-market) ──► для каждого агента:
     1. Data Collector  — цены, история, индикаторы, новости → брифинг
     2. Agent Core      — Claude API, structured output → предложения сделок
     3. Guardrails Engine — детерминированный фильтр (НЕ LLM), дёшево, первым
     4. Validation Panel  — 2–3 LLM-валидатора с разными ролями, только вето
            мелкие → авто-исполнение
            крупные → подтверждение в Telegram (кнопка)
     5. Broker Adapter  — ib-async → IBKR Gateway → paper account
     6. Ledger + Decision Journal (Postgres/SQLite)
     7. Reporter        — Telegram-бот: дайджест, алерты, подтверждения
  Watchdog (независимый процесс) — следит за NAV, глобальный стоп
```

### Компоненты (изолированные модули)

1. **Data Collector** — собирает рыночные данные (цены/история), считает индикаторы,
   подтягивает новости, формирует структурированный «брифинг» для агента.
2. **Agent Core** — на вход брифинг + текущие позиции агента; на выход структурированные
   предложения сделок с обоснованием (Claude API, strict structured output).
3. **Guardrails Engine** — детерминированный (не LLM) фильтр: валидация, лимиты,
   kill-switch, whitelist, порог подтверждения. Тестируется независимо.
4. **Validation Panel** — adversarial-слой из 2–3 LLM-валидаторов с разными ролями
   (см. §5.1). Только вычитающий: может наложить вето или урезать размер, но не может
   добавить сделку или ослабить лимиты. Конфигурируемый, можно выключить.
5. **Broker Adapter** — обёртка над IBKR (ib-async): исполнение, котировки, состояние счёта.
6. **Ledger + Decision Journal** — БД: предложения, обоснования, исполнения, результаты.
7. **Reporter** — Telegram-бот: дайджесты, алерты, кнопки подтверждения.
8. **Scheduler** — cron, запускает дневной цикл pre-market.
9. **Watchdog** — независимый процесс контроля NAV и глобального стопа.

## 5. Движок решений и исполнение

- Claude API вызывается раз в день на агента, получает структурированный брифинг и
  текущие позиции, возвращает **только** список вида
  `{действие, тикер, количество, стоп-лосс}` через strict schema. Палитра действий
  ограничена схемой — вывод денег, опционы и прочее физически невыразимы.
- **Контроль исполнения:** сделки ниже порога ($500 или 25% бюджета) исполняются авто;
  крупные ждут подтверждения кнопкой в Telegram. Механизм подтверждения строится сразу
  (он же нужен для реала).
- **Обучение/коррекция:** журнал решений хранит предложение + обоснование + результат.
  Раз в неделю Claude получает свои прошлые результаты и корректирует подход
  (self-review). Полноценная ML-прогнозная модель — отдельный будущий проект.

### 5.1 Validation Panel (adversarial-валидация)

Слой поверх детерминированных guardrails, ловит «Класс 2» ошибок LLM — правдоподобные,
но неверные решения, которые формально проходят все жёсткие лимиты. Порядок не случаен:
сначала дешёвый детерминированный фильтр режет хард-нарушения, и только выжившие сделки
уходят в дорогую LLM-панель — так тратится меньше токенов.

**Ключевая оговорка — коррелированные ошибки.** Несколько одинаковых вызовов Claude
ошибаются одинаково (одна модель, одно обучение). Поэтому валидаторы обязаны быть
**ролево-разнообразными**, каждый со своим углом атаки:
- **Риск-скептик:** где даунсайд, что если тезис неверен, адекватен ли стоп.
- **Проверка катализаторов:** нет ли известного события (отчёт, дивотсечка, макро),
  делающего сделку глупой именно сегодня.
- **Адвокат дьявола:** активно пытается опровергнуть тезис сделки.

**Два жёстких правила:**
1. **Только вычитающий (subtractive).** Панель может наложить вето или урезать размер,
   но не может добавить сделку или ослабить лимиты. Только режет.
2. **Польза измеряется на бумаге.** Каждое вето логируется вместе с гипотетическим
   исходом (что было бы, если пропустить). За 6 месяцев оценивается, окупается ли панель;
   если нет — выключается (она конфигурируема).

**Строгость вето — по профилю риска:**
| Профиль | Правило блокировки |
|---|---|
| Консервативный | любое вето → блок |
| Умеренный | большинство против → блок |
| Агрессивный | большинство против → блок |

## 6. Управление риском (защита капитала)

### Риск А — деньги уходят со счёта
Торговый API IBKR не умеет выводить средства; вывод — отдельный канал с отдельной
аутентификацией на привязанный банковский счёт. Кража через торговый API закрыта
самим брокером.

### Риск Б — потери на сделках. Слои защиты:

- **Слой 0 — отдельный счёт с ограниченным фондированием (главный).** При переходе на
  реал агент подключается к отдельному linked-счёту IBKR, куда переведена только
  песочница (~$3k). До основных $30k агент физически не дотягивается — это граница на
  стороне брокера, а не лимит в коде.
- **Слой 1 — Precautionary Limits в IBKR Gateway:** брокер сам отклоняет ордера сверх
  настроенного размера/стоимости.
- **Слой 2 — Guardrails Engine:** бюджет на агента, лимит на позицию, порог
  авто-исполнения, обязательный стоп-лосс, whitelist инструментов, лимит сделок/день.
- **Слой 3 — Kill-switches** (см. таблицу ниже).
- **Слой 4 — Watchdog:** независимый процесс смотрит реальный NAV; при пробое пола —
  глобальный стоп (отмена ордеров, закрытие позиций, заморозка агентов, алерт).
- **Слой 5 — Reconciliation:** каждый цикл сверяет ledger с реальным счётом; расхождение
  или неизвестная позиция → заморозка + алерт.

### Защита от ошибок LLM

- **Класс 1 (бред/мусор):** strict schema + поштучная валидация каждого поля (тикер
  существует и в whitelist; количество × цена в пределах бюджета и лимита; стоп-лосс
  присутствует; названная LLM цена совпадает с рыночной в пределах допуска). Не прошло —
  reject + лог + алерт; валидные сделки проходят.
- **Класс 2 (правдоподобное, но неверное):** Validation Panel (§5.1) как
  adversarial-фильтр + ограничение ущерба — лимит на позицию, обязательный стоп-лосс,
  диверсификация, дневной и просадочный kill-switch.
- Худший исход ошибки LLM — заранее посчитанный ограниченный убыток внутри песочницы,
  не катастрофа.

### Kill-switches

| Профиль | Дневной убыток → стоп на день | Просадка от пика → полная остановка |
|---|---|---|
| Консервативный | −3% | −10% |
| Умеренный | −5% | −15% |
| Агрессивный | −8% | −25% |

### Глобальные предохранители

- Авто-исполнение только сделок < $500 (или < 25% бюджета); крупнее → подтверждение в Telegram.
- Watchdog: общий NAV < 80% стартового → глобальный стоп.
- Reconciliation: расхождение ledger ↔ счёт → заморозка + алерт.

## 7. Отчётность

Telegram-бот:
- **Утренний дайджест:** что каждый агент решил и почему, какие сделки прошли/отклонены.
- **Уведомление по каждой сделке.**
- **Запрос подтверждения** крупных сделок (inline-кнопки).
- **Алерты:** срабатывание guardrails, kill-switch, reconciliation, watchdog.
- **Периодический P&L-отчёт** по каждому агенту и суммарно.

## 8. Деплой и инфраструктура

- **VPS:** Linux (Ubuntu), ~2 vCPU / 4 GB. Hetzner / DigitalOcean / Vultr. ~$10–20/мес.
- **IBKR Gateway (headless):** держит сессию с IBKR; агент подключается по локальному
  сокету. Авто-логин и суточный перезапуск — через IBC (IBController). 2FA для
  headless-доступа настраивается на этапе внедрения. Paper↔Real — переключение флагом.
- **Упаковка:** docker-compose:
  - `ib-gateway` (IBC + Gateway),
  - `app` (Python: агенты, guardrails, broker adapter, scheduler, reporter, watchdog),
  - `db` (Postgres; либо SQLite-том для простого варианта).
- **Telegram-бот** — постоянный процесс в `app`.
- **Развёртывание:** (1) VPS + Docker; (2) `.env` с секретами + `docker-compose up -d`;
  (3) разовая проверка логина Gateway и видимости paper-счёта.
- **Секреты:** только в `.env` на VPS (не в git), доступ по SSH-ключу, файрвол закрывает
  всё кроме SSH, API-порт Gateway слушает только localhost.
- **Состояние:** ledger и журнал в томе БД, бэкапятся.

## 9. Технологический стек

- **Язык:** Python.
- **IBKR:** ib-async (форк ib_insync) + IB Gateway + IBC.
- **LLM:** Anthropic Claude API (structured output).
- **БД:** Postgres (или SQLite на старте).
- **Бот:** Telegram Bot API.
- **Планировщик:** cron.
- **Оркестрация:** Docker / docker-compose.

## 10. Ожидания по доходности

Гарантий нет. Бенчмарк — SPY (~7–10%/год в долгую), который обыгрывают <10–20%
профессиональных фондов на горизонте 10+ лет. У LLM-торговли нет доказанного
преимущества. Реалистичное распределение исходов: от −10…−20% (вероятно) до обгона
рынка на несколько процентов (отличный результат). Доходность съедают комиссии, спред,
проскальзывание и стоимость токенов. Три профиля — это про разброс, а не про
«больше риск = больше прибыль».

## 11. Бэктестинг, lookahead bias и форвард-тест

**Решения LLM не бэктестятся на истории — никогда.** Две причины утечки будущего
(lookahead bias):

1. **Training cutoff.** Модель обучена на данных до даты отсечки и для любой исторической
   даты «знает» исход. «Предсказание» становится воспоминанием; бэктест даёт фиктивно
   отличные результаты, которые рассыпаются на реальных деньгах.
2. **Веб-поиск.** На исторической дате поиск буквально находит исход — прямое жульничество.

Следствия для дизайна:
- **LLM-стратегия валидируется только форвард-симуляцией** — 6 месяцев на бумаге в
  реальном времени, где будущее не наступило ни для владельца, ни для модели. Именно
  поэтому ворота go-live построены на форвард-тесте, а не на бэктесте.
- **Бэктестить можно только детерминированные части** (индикаторы, логика guardrails) —
  они механические, без «памяти». Прогон строго point-in-time: Data Collector обязан
  отдавать данные «как на тот день», без подглядывания вперёд.
- **В форвард-режиме веб-поиск разрешён и полезен** — будущего ещё нет; поиск свежих
  новостей утром торгового дня и есть работа аналитика. Утечка — проблема только
  исторических дат.
- Еженедельный self-review честен: Claude разбирает свои собственные форвард-решения,
  а не подсматривает рынок.

## 12. Ворота go-live (переход на реальные деньги)

Период симуляции — **6 месяцев** на paper-счёте. Реал подключается, только если:

1. агент обогнал SPY по риск-скорректированной доходности (Sharpe) с учётом комиссий и
   проскальзывания, И
2. максимальная просадка не вышла за лимит профиля.

Старт реала — отдельный счёт IBKR на ~$3k, не основные $30k.

## 13. Вынесено за рамки MVP (YAGNI)

- ML-прогнозная модель (отдельный проект).
- Веб-дашборд (на старте — только Telegram).
- Опционы, фьючерсы, плечо, торговля чаще раза в день.

## 14. Критерии успеха

- Система работает на VPS 24/7, ежедневно прогоняет цикл всех трёх агентов на бумаге.
- Все слои guardrails и kill-switch покрыты тестами и срабатывают детерминированно.
- Владелец получает в Telegram дайджесты, алерты и подтверждает крупные сделки.
- За 6 месяцев накоплен журнал решений с результатами для оценки ворот go-live.
- Ни один баг или ошибка LLM не приводит к потере сверх заранее заданных лимитов.

## 15. Порядок реализации (черновик)

Детальный план — на следующем шаге (writing-plans). Ориентировочные вехи:

1. Каркас проекта, конфиг профилей, БД-схема (ledger + journal).
2. Broker Adapter + подключение к IBKR paper через Gateway/IBC.
3. Guardrails Engine (детерминированный, с тестами) — сердце безопасности.
4. Data Collector + брифинг.
5. Agent Core (Claude API, structured output) + валидация предложений.
6. Validation Panel (ролевые валидаторы, вето по профилям, лог гипотетических исходов).
7. Reporter (Telegram: дайджест, алерты, подтверждения).
8. Watchdog + reconciliation.
9. Scheduler + сборка docker-compose + деплой на VPS.
10. Прогон на бумаге, метрики, оценка ворот go-live.
